The issue — US automation hit a Meta rate limit

Meta's Marketing API enforces a per-user request limit shared across every ad account on a single access token. After running ~150 API operations earlier in the session (the UK batch of changes plus their verification reads, then the four other markets), the token reached that limit at the point we tried to load the US account.

Why the US account specifically

  • It's the largest account. Loading its ad set list returns the heaviest data payload of any market — so it's the first request Meta refuses when the limit is close, and the last it allows back.
  • The limit is token-wide, not account-specific. The earlier markets had already "spent" the token's request budget for the window; the US load was simply the request that tipped it over.
  • Recovery is gradual. Meta's limit refills over time only if the token goes quiet. Because we were actively completing the other markets, the counter kept topping up.
ℹ️ Important context

This is a platform throttle, not an error or a failure in our system. It's Meta protecting its own infrastructure. It affects how fast we can push automated changes in a single session — it does not affect ad performance, billing, or the correctness of any decision.

How it was handled

UK + 4 markets deployed automatically
100 actions executed via CLI, each previewed, confirmed, executed one-at-a-time, then re-read from Meta to verify the change landed. 0 failures.
US load refused by rate limit
Two timed retries (25 min apart) plus a clean cooldown attempt — limit persisted because the token stayed active completing other markets.
Switched US to manual deployment
Generated a full checklist of all 18 US ad sets — each with the rule that triggered it, last-week performance, and the exact budget action. Deployed by hand in Ads Manager.
Outcome: 100% of approved changes live
Same decisions, same rule set, fully deployed. The only difference for the US set is the deployment route.

Preventing it next time

The fix is operational, not structural — we slow down and lighten the API footprint so a single session never trips the limit:

  • Throttle & sequence markets. Run accounts one at a time with deliberate pauses between, rather than in parallel, so the token's request budget never spikes.
  • Lighter data requests. Request only the fields we need (status, budget) instead of the full ad set record, dramatically reducing payload weight on large accounts like the US.
  • Largest account first. Load the US while the token's request budget is full, before the smaller markets consume it.
  • Pace the heavy verification reads so the post-change checks don't compete with the next market's load.
Net

With these adjustments, next week's run should complete all markets — US included — automatically end-to-end. The manual fallback remains available and proved reliable.

Governance & safety

Every automated change this session passed the agreed write-safety controls: no deletions (pause only, fully reversible), a full preview and typed confirmation before execution, a bulk-operation gate for any batch touching more than 20% of a market's live ad sets, a single low-risk pilot change verified before each bulk run, and an audit-log entry recording the before/after state of every change.
